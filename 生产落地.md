# 生产落地待办

本文档用于记录当前项目从开发/测试环境落到生产环境的改造方案、准备清单与执行步骤。当前阶段只作为参考待办，不涉及代码修改。

## 一、目标

- Client 以 Windows EXE 安装包形式部署在大陆用户电脑。
- SM 部署在稳定境外云服务器，作为认证、节点管理、配置管理、占用控制的控制面。
- TS 部署在多台境外云服务器，作为实际交易执行节点。
- Client 只配置 SM 地址，不固定写入 TS IP。
- Client 登录 SM 后，由 SM 分配可用 TS。
- Client 与 TS 使用 WSS 长连接。
- TS 同一时间只服务一个 Client，保持独占模型。
- TS 地址或服务器变化时，不要求 Client 重新安装或手动改配置。

## 四、代码现状与需要改造的点

### 1. Client HTTP 地址配置

当前：

- `Client/network/http_client.py` 通过 `host + port` 拼接 `http://host:port`。
- `Client/constants.py` 使用 `CLIENT_SM_HOST` 和 `CLIENT_SM_PORT`。

待改造：

- 支持完整 SM URL，例如 `https://sm.yourdomain.com`。
- 建议新增环境变量或安装包配置项：
  - `CLIENT_SM_BASE_URL=https://sm.yourdomain.com`
- 保留旧的 `CLIENT_SM_HOST/CLIENT_SM_PORT` 作为兼容 fallback。

### 2. Client 到 TS 的 WebSocket

当前：

- `Client/network/ts_websocket.py` 固定拼接 `ws://{host}:{port}/ws`。
- `Client/ui_qt/main_window.py` 将 `se_address` 拆成 host/port 后连接。

待改造：

- 支持完整 WebSocket URL：
  - `wss://sg-01.ts.yourdomain.com/ws`
  - `ws://127.0.0.1:8900/ws`
- SM 分配 TS 后直接返回 `wss_url`。
- Client 使用该 `wss_url` 建立连接。

### 3. SM 分配 TS

当前：

- Client 登录后拿到数据库账号上的 `se_address`。
- Client 再通过 `/api/accounts/se-status?address=...` 查找匹配节点。
- 匹配主要依赖 `host/current_ip/address` 字符串。

待改造：

- 新增生产主链路接口：
  - `POST /api/trading/session/allocate`
- 接口职责：
  - 校验 Client token。
  - 根据账号权限、券商类型、节点在线状态、占用状态、区域策略选择 TS。
  - 返回 `server_id + wss_url + node_name + region`。
- 旧接口 `/api/accounts/se-status?address=...` 保留为兼容或诊断接口。

建议返回格式：

```json
{
  "ok": true,
  "server_id": "ts-sg-01",
  "node_name": "Singapore TS 01",
  "region": "SG",
  "broker_type": "tastytrade",
  "wss_url": "wss://sg-01.ts.yourdomain.com/ws",
  "expires_in": 60
}
```

### 4. TS 注册与心跳

当前：

- `Trader_Server/services/registration.py` 注册时提交 `node_name/region/host/capabilities/contact/description`。
- `Trader_Server/services/heartbeat.py` 心跳上报 `ip`，SM 记录 `current_ip`。

待改造：

- TS 注册和心跳需要明确上报公网入口：
  - `public_ws_url`
  - `public_http_url`
  - `current_ip`
  - `region`
  - `broker_type`
  - `node_name`
- SM 数据库需要保存这些字段。

建议 TS 环境变量：

```text
TS_MANAGER_URL=https://sm.yourdomain.com
TS_NODE_NAME=ts-sg-01
TS_REGION=SG
TS_BROKER_TYPE=tastytrade
TS_PUBLIC_WS_URL=wss://sg-01.ts.yourdomain.com/ws
TS_PUBLIC_HTTP_URL=https://sg-01.ts.yourdomain.com
TS_WS_PORT=8900
```

### 5. SM 数据库字段

当前已有：

- `accounts`
- `nodes`
- `node_runtime`
- `node_broker_config`
- `node_requests`

待补充字段：

- `nodes.public_ws_url`
- `nodes.public_http_url`
- `nodes.region`
- `node_runtime.current_ip`
- 视需要增加 `node_runtime.last_seen_public_ip`

保留：

- `host` 字段可作为兼容字段。

### 6. Token 与 Session

当前：

- Client token 在 SM 内存 `active_client_tokens` 中。
- Admin session 也在 SM 内存中。
- SM 重启后，Client 登录状态丢失。

第一阶段可以接受：

- 单 SM 实例。
- 重启后要求 Client 重新登录。

正式生产建议：

- Client token/session 放 Redis 或数据库。
- Admin session 放 Redis 或数据库。
- 节点占用状态可以继续内存优先，但需要更可靠的恢复策略。

### 7. HTTPS/WSS 入口

SM：

```text
Client
  |
  | https://sm.yourdomain.com
  v
SM Nginx/Caddy :443
  |
  | http://127.0.0.1:8800
  v
Server_manager
```

TS：

```text
Client
  |
  | wss://sg-01.ts.yourdomain.com/ws
  v
TS Nginx/Caddy :443
  |
  | ws://127.0.0.1:8900/ws
  v
Trader_Server
```

要求：

- 每台 TS 都部署自己的 Nginx/Caddy。
- 不使用中央中继，避免额外跨公网延迟。
- 服务器只对公网开放 443。
- TS 内部 8900 只监听本机或只允许本机访问。

## 五、你需要准备的东西

### 1. 域名

- 一个主域名，例如 `yourdomain.com`。
- SM 子域名：
  - `sm.yourdomain.com`
- TS 子域名规则：
  - `sg-01.ts.yourdomain.com`
  - `sg-02.ts.yourdomain.com`
  - `jp-01.ts.yourdomain.com`
  - `us-01.ts.yourdomain.com`

### 2. DNS

- DNS 服务商账号。
- 最好支持 API，方便后续 TS IP 变化后自动更新 DNS。
- TS 子域名 TTL 建议设置为 60 秒。

DNS 示例：

```text
sm.yourdomain.com          -> SM 服务器公网 IP
sg-01.ts.yourdomain.com    -> 新加坡 TS 01 公网 IP
sg-02.ts.yourdomain.com    -> 新加坡 TS 02 公网 IP
jp-01.ts.yourdomain.com    -> 日本 TS 01 公网 IP
```

### 3. 证书

- `sm.yourdomain.com` HTTPS 证书。
- `*.ts.yourdomain.com` 通配符证书。

推荐：

- Caddy 自动申请和续期。
- 或 Nginx + Certbot + DNS 插件。

### 4. 云服务器

SM 服务器：

- 境外稳定 IP，建议新加坡。
- 开放 443。
- 部署 SM。
- 部署 Nginx/Caddy。
- 准备数据库和备份策略。

TS 服务器：

- 可多地区部署。
- 每台开放 443。
- 每台部署 TS。
- 每台部署 Nginx/Caddy。
- 每台使用自己的子域名。

### 5. 安全配置

- 不把 Broker 凭据放入 Client。
- 不在日志中输出 Broker 密码、token、secret。
- SM 管理后台使用强密码。
- 生产关闭测试后门账号。
- 防火墙只开放必要端口。
- 备份 SM 数据库。

### 6. Client 安装包配置

Client 安装包只需要内置或安装后写入：

```text
CLIENT_SM_BASE_URL=https://sm.yourdomain.com
```

不要写入：

- TS IP
- TS 固定地址
- Broker 凭据
- Broker token

## 八、风险与决策点

### 1. 是否第一阶段引入 Redis/PostgreSQL

建议：

- 最小生产可用阶段可以先用单 SM + SQLite。
- 只要接受 SM 重启后 Client 需要重新登录即可。
- 正式生产或多 SM 时再引入 Redis/PostgreSQL。

### 2. TS IP 是否频繁变化

如果只是偶尔换服务器或 IP：

- 子域名 + DNS 低 TTL 足够。

如果 TS IP 非常频繁变化：

- 需要 DNS API 自动更新。
- 或考虑反向隧道/中继，但会增加架构复杂度和延迟。

### 3. 是否使用中央中继

当前不建议。

推荐：

- Client 直接 WSS 连接被 SM 分配的 TS。
- 每台 TS 自己有 Nginx/Caddy。

不推荐第一阶段：

- Client -> 中央中继 -> TS。

原因：

- 增加跨公网转发延迟。
- 增加单点风险。
- 改造范围更大。

### 4. 生产测试实盘风险

- Broker 登录可以测试。
- 查询持仓、订单可以测试。
- 下单、撤单测试必须单独确认范围。
- 实盘账户禁止在自动化测试中误提交订单。
