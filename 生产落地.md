# 生产落地

本文档用于把当前三端项目从本地/测试环境落到生产环境。当前只整理方案和待办，不修改业务代码。

本轮已复核三端通信相关代码与打包脚本，范围包括：

- Client：`Client/main.py`、`Client/constants.py`、`Client/config.py`、`Client/network/*`、`Client/services/*`、`Client/ui_qt/*`、`Client/ui/*`
- SM：`Server_manager/main.py`、`Server_manager/config.py`、`Server_manager/auth.py`、`Server_manager/database.py`、`Server_manager/node_state.py`、`Server_manager/models.py`、`Server_manager/routers/*`、`Server_manager/services/*`、`Server_manager/templates/*`
- TS：`Trader_Server/main.py`、`Trader_Server/config.py`、`Trader_Server/models.py`、`Trader_Server/network/*`、`Trader_Server/services/*`、`Trader_Server/api/*`、`Trader_Server/ui_qt/*`
- 打包脚本：`build_client.bat`、`build_sm.bat`、`build_ts.bat`

## 一、总纲

### 1. 生产目标

- Client 部署在大陆地区 Windows 电脑，通过 EXE 安装/运行。
- SM 部署在境外稳定 Windows 云服务器，承担认证、账号、节点、配置、占用控制、管理后台职责。
- TS 部署在境外 Windows 云服务器，可多台、多地区，承担实际券商登录、持仓、订单、行情、下单、撤单职责。
- Client 先登录 SM，SM 返回当前交易员账号绑定的 TS 地址，Client 再连接该 TS。
- 不做 TS 动态分配。继续保持“交易员账号绑定指定 TS”的现有逻辑。
- 生产正式链路使用域名 + HTTPS/WSS：
  - Client -> SM：`https://sm.yourdomain.com`
  - Client -> TS：`wss://sg-01.ts.yourdomain.com/ws`
- 生产第一版使用 Caddy 自动申请和续期证书。WSS 不需要单独证书，WSS 使用的就是 HTTPS/TLS 证书。
- Caddy 不放在 Client 上。SM 服务器和每台 TS 服务器各自部署一份 Caddy。
- Caddy 不是中央中继，只是每台服务器本机的公网入口和反向代理。
- SM 和 TS 的 FastAPI 内部端口不直接暴露公网，公网只开放必要入口。
- TS 同一时间只服务一个 Client，保持现有独占模型。
- TS 首次注册生成的 `Trader_Server/data/config.json` 继续放在当前本地路径，暂不做路径迁移。
- Broker secret/token 放在 SM 是当前正确方向，第一版暂不做数据库加密。
- 第一版按实盘真实提交落地，不做 dry-run/只读交易开关。

### 3. 已确认原则

- 只做 Windows EXE 生产落地，不做 Linux/systemd/headless 部署。
- 不做动态分配 TS，不按区域、延迟、负载自动选择 TS。
- 账号绑定 TS 支持 IP、域名、完整 WSS URL 三种输入。
- 生产推荐在 SM 账户绑定中保存完整 WSS URL，例如 `wss://sg-01.ts.yourdomain.com/ws`。
- `region` 字段当前实际承载券商类型，第一版不改名，只在文档和 UI 表述中避免把它当地理区域使用。
- Client token/session 当前存 SM 内存，第一版不做 Redis/DB 持久化。
- SM 重启后 Client 重新登录可以接受。
- 本地敏感文件、默认账号、后门账号放到最后打包前统一处理。
- SM 不承担券商行情链路。行情后续全部走 Client -> TS -> Broker。
- 旧 SM 行情代码先谨慎关闭，确认无依赖后再删除。
- 密码强度升级、依赖锁定、数据库内券商凭据加密后续再做。

### 4. 证书与 Caddy 结论

- 每个对外域名都需要证书：
  - `sm.yourdomain.com`
  - `sg-01.ts.yourdomain.com`
  - `sg-02.ts.yourdomain.com`
- 第一版不强制通配符证书。每台服务器上的 Caddy 自动给本机域名申请证书即可。
- 证书通常有效期较短，Caddy 会自动提前续期。只要 Caddy 数据目录保留、80/443 可访问、DNS 指向正确，一般不需要人工频繁更换。
- 不要删除 Caddy 的数据目录，否则可能触发重新申请证书，频繁重申可能遇到 CA 限流。
- 后续 TS 数量很多时，再考虑 `*.ts.yourdomain.com` 通配符证书和 DNS API 自动化。
- Caddy 可以随 SM/TS 安装包一起分发 `caddy.exe`，但不要打进 PyInstaller 生成的业务 EXE 内部。
- 每台服务器首次部署时配置一次 Caddyfile。后续只升级 SM/TS EXE 时，通常不需要重新配置 Caddy，除非域名、端口或路由规则改变。

### 8. 临时占位与域名切换策略

域名未申请好之前，可以先使用 IP 占位继续测试和使用，但要区分“功能测试”和“生产安全链路测试”：

```text
本机测试占位:
  CLIENT_SM_BASE_URL=http://127.0.0.1:8800
  TS 地址=127.0.0.1:8900
  或 TS 地址=ws://127.0.0.1:8900/ws

局域网/公网 IP 测试占位:
  CLIENT_SM_BASE_URL=http://<SM_IP>:8800
  TS 地址=<TS_IP>:8900
  或 TS 地址=ws://<TS_IP>:8900/ws

生产域名占位:
  SM_DOMAIN=sm.yourdomain.com
  TS_DOMAIN_01=sg-01.ts.yourdomain.com
  TS_WSS_01=wss://sg-01.ts.yourdomain.com/ws
```

结论：

- 当前为了保障测试和临时使用，可以用本机 IP 或公网 IP 占位。
- IP 占位建议走 `http://` 和 `ws://`，用于验证业务链路、登录、占用、Broker API、持仓、订单、行情。
- 不建议把裸 IP + WSS 当作第一版正式方案。证书、浏览器/客户端信任、Caddy 自动申请都会比域名方式麻烦，且不符合最终生产目标。
- 如果域名没申请好但必须提前测试 HTTPS/WSS，优先使用一个临时子域名指向服务器 IP，而不是裸 IP。
- 后续域名申请好后，理想情况下只改配置，不改业务逻辑：
  - Client 改 `CLIENT_SM_BASE_URL`
  - SM 后台交易员账号改绑定的 TS WSS 地址
  - TS 注册 host 改为对应 TS 域名
  - Caddyfile 中替换占位域名

### 9. 暂不做项目

- TS 动态分配。
- Redis/DB 保存 Client token/session。
- 数据库内 Broker credentials 加密。
- 密码强度升级。
- requirements 依赖锁定。
- dry-run/只读交易开关。
- Linux/systemd/headless 部署。
- 真正的地理字段 `deployment_region`。
- DNS API 自动更新。
- 通配符证书。

### 10. 需要你准备

- 一个主域名，例如 `yourdomain.com`。
- SM 子域名，例如 `sm.yourdomain.com`。
- 每台 TS 一个子域名，例如：
  - `sg-01.ts.yourdomain.com`
  - `sg-02.ts.yourdomain.com`
  - `jp-01.ts.yourdomain.com`
- 境外 Windows 云服务器：
  - 1 台 SM，IP 尽量稳定。
  - N 台 TS，IP 可以变，变更后更新 DNS。
- 每台服务器开放 80/443，关闭公网 8800/8900。
- 确认 Caddy 数据目录位置，部署后不要随意删除。
- 确认 SM 后台生产管理员账号。
- 确认每个交易员账号要绑定哪台 TS。
