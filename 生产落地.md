## 一、总纲

### 1. 生产目标

- Client 部署在大陆地区 Windows 电脑，通过 EXE 安装/运行。
- SM 部署在境外稳定 Windows 云服务器，承担认证、账号、节点、配置、占用控制、管理后台职责。
- TS 部署在境外 Windows 云服务器，可多台、多地区，承担实际券商登录、持仓、订单、行情、下单、撤单职责。
- Client 先登录 SM，SM 返回当前交易员账号绑定的 TS 地址，Client 再连接该 TS。
- 不做 TS 动态分配。继续保持“交易员账号绑定指定 TS”的现有逻辑。
- SM 托管一批 TS 子域名池，并通过腾讯云 DNSPod API 自动把被分配的子域名解析到注册 TS 上报的公网 IP。
- 子域名池分配只解决“TS 使用哪个公网域名”，不改变“交易员账号绑定指定 TS”的业务模型。
- 生产正式链路使用域名 + HTTPS/WSS：
  - Client -> SM：`https://scjrdomain.com`
  - Client -> TS：`wss://ts-01.ts.scjrdomain.com/ws`
- 生产第一版使用 Caddy 自动申请和续期证书。WSS 不需要单独证书，WSS 使用的就是 HTTPS/TLS 证书。
- Caddy 不放在 Client 上。SM 服务器和每台 TS 服务器各自部署一份 Caddy。
- Caddy 不是中央中继，只是每台服务器本机的公网入口和反向代理。
- SM 和 TS 的 FastAPI 内部端口不直接暴露公网，公网只开放必要入口。
- TS 同一时间只服务一个 Client，保持现有独占模型。
- TS 首次注册生成的 `Trader_Server/data/config.json` 继续放在当前本地路径，暂不做路径迁移。
- TS 未注册时启动注册流程必须上报本机当前公网 IP；已注册时不自动重走注册。
- TS 已注册后公网 IP 变化，第一版暂不做心跳自动更新 DNS，采用人工清理旧注册/重新注册或人工调整 DNS 的方式处理。
- Broker secret/token 放在 SM 是当前正确方向，第一版暂不做数据库加密。
- 第一版按实盘真实提交落地，不做 dry-run/只读交易开关。

### 3. 已确认原则

- 只做 Windows EXE 生产落地，不做 Linux/systemd/headless 部署。
- 不做动态分配 TS，不按区域、延迟、负载自动选择 TS。
- 做 SM 子域名池自动分配：TS 注册审批通过时，SM 从域名池中选取未占用域名并调用腾讯云 DNSPod API 修改解析。
- 子域名池状态至少包含：未占用、已占用、冷却中、异常。冷却中使用蓝色 tag 展示。
- 域名池“占用者”定义为 TS 节点名称，并附带该 TS 上报的占用 IP；Client 占用者仍使用节点状态中的 `occupied_by` 单独展示。
- SM 删除某个 TS 时，不检查交易员账号是否仍绑定该 TS 域名；删除后及时释放域名池占用信息并进入冷却，旧交易员账号如未改绑则由 Client 连接失败暴露问题。
- TS 已注册后公网 IP 变化，第一版不做自动 DNS 修正；需要人工清理旧注册后重新注册，或人工调整 DNS。
- 账号绑定 TS 支持 IP、域名、完整 WSS URL 三种输入。
- 生产推荐在 SM 账户绑定中保存完整 WSS URL，例如 `wss://ts-01.ts.scjrdomain.com/ws`。
- 生产推荐在 SM 创建/编辑交易员账号时，通过下拉选择“已被 TS 占用的域名”，保存其完整 WSS 地址；保留手动输入作为本地/IP 占位测试兜底。
- TS 未注册且支持注册时，启动运行后自动获取当前设备公网 IP，并在注册申请中强制上报该 IP；已注册 TS 不重复执行该获取步骤。
- `region` 字段当前实际承载券商类型，第一版不改名，只在文档和 UI 表述中避免把它当地理区域使用。
- Client token/session 当前存 SM 内存，第一版不做 Redis/DB 持久化。
- SM 重启后 Client 重新登录可以接受。
- 本地敏感文件、默认账号、后门账号放到最后打包前统一处理。
- SM 不承担券商行情链路。行情后续全部走 Client -> TS -> Broker。
- 旧 SM 行情代码先谨慎关闭，确认无依赖后再删除。
- 密码强度升级、依赖锁定、数据库内券商凭据加密后续再做。

### 4. 证书与 Caddy 结论

- 每个对外域名都需要证书：
  - `scjrdomain.com`
  - `ts-01.ts.scjrdomain.com`
  - `ts-02.ts.scjrdomain.com`
- 第一版不强制通配符证书。每台服务器上的 Caddy 自动给本机域名申请证书即可。
- 证书通常有效期较短，Caddy 会自动提前续期。只要 Caddy 数据目录保留、80/443 可访问、DNS 指向正确，一般不需要人工频繁更换。
- 不要删除 Caddy 的数据目录，否则可能触发重新申请证书，频繁重申可能遇到 CA 限流。
- DNSPod API 自动更新解析由 SM 子域名池功能负责；后续 TS 数量很多时，再考虑 `*.ts.scjrdomain.com` 通配符证书。
- Caddy 自动化部署代码已完成第一版；真实 HTTPS/WSS 证书验收仍需在 SM 和 TS 生产服务器上配置 `caddy.exe`、开放 80/443 并完成实机验证。
- Caddy 可以随 SM/TS 安装包一起分发 `caddy.exe`，但不要打进 PyInstaller 生成的业务 EXE 内部。
- 每台服务器首次部署时配置一次 Caddyfile。后续只升级 SM/TS EXE 时，通常不需要重新配置 Caddy，除非域名、端口或路由规则改变。


结论：

- 当前为了保障测试和临时使用，可以用本机 IP 或公网 IP 占位。
- IP 占位建议走 `http://` 和 `ws://`，用于验证业务链路、登录、占用、Broker API、持仓、订单、行情。
- 不建议把裸 IP + WSS 当作第一版正式方案。证书、浏览器/客户端信任、Caddy 自动申请都会比域名方式麻烦，且不符合最终生产目标。
- 如果域名没申请好但必须提前测试 HTTPS/WSS，优先使用一个临时子域名指向服务器 IP，而不是裸 IP。
- 后续域名申请好后，理想情况下只改配置，不改业务逻辑：
  - Client 改 `CLIENT_SM_BASE_URL`
  - SM 子域名池录入可分配的 TS 子域名
  - SM 后台交易员账号从已占用域名下拉选择绑定的 TS WSS 地址
  - TS 注册时上报公网 IP，由 SM 分配对应 TS 域名
  - Caddyfile 中使用 SM 分配给该 TS 的域名

### 9. 暂不做项目

- TS 动态分配。
- Redis/DB 保存 Client token/session。
- 数据库内 Broker credentials 加密。
- 密码强度升级。
- requirements 依赖锁定。
- dry-run/只读交易开关。
- Linux/systemd/headless 部署。
- 真正的地理字段 `deployment_region`。
- 已注册 TS 公网 IP 变化后的心跳自动 DNS 修正。
- 删除 TS 前检查交易员账号是否仍绑定该 TS 域名。
- 通配符证书。

### 10. 需要你准备

- 一个主域名，例如 `scjrdomain.com`。
- SM 正式 Web/API 域名：`scjrdomain.com`。
- 一批 TS 子域名池，例如 20 个：
  - `ts-01.ts.scjrdomain.com`
  - `ts-02.ts.scjrdomain.com`

  - N 台 TS，注册时必须能获取或填写当前公网 IP；IP 变化时第一版人工重新注册或人工调整 DNS。
