## 1. 产品概述

### 1.1 产品背景

随着证券交易市场的快速发展，交易员对交易系统的稳定性、实时性和便捷性提出了更高要求。传统基于HTTP轮询的交易系统存在延迟高、资源消耗大、实时性差等问题，无法满足专业交易员的需求。

本系统采用WebSocket长连接技术，构建一套高效、稳定、实时的证券股票交易平台，实现交易员与券商交易系统的实时双向通信。

### 1.2 产品目标

- **高效性**：实现毫秒级订单响应，提升交易效率
- **稳定性**：长连接保活机制，确保连接稳定
- **安全性**：多重安全验证，保障交易安全
- **可扩展性**：模块化设计，支持功能扩展
- **可维护性**：完善的监控运维体系

### 1.3 目标用户

| 用户类型 | 角色描述 | 主要功能 |
|---------|---------|---------|
| 交易员 | 执行交易操作 | 行情查看、下单、撤单、持仓查询 |
| 公司管理员 | 管理交易员账户 | 账户管理、权限分配、券商管理 |
| 系统运维 | 系统维护监控 | 状态监控、故障处理、日志分析 |

## 4. 技术方案

### 4.1 WebSocket长连接技术方案

#### 4.1.1 方案概述

本系统采用WebSocket长连接作为客户端与服务端之间的通信方式。相比传统的HTTP轮询，WebSocket具有以下优势：

| 特性 | HTTP轮询 | WebSocket长连接 |
|------|---------|----------------|
| 响应延迟 | 高（需等待轮询） | 低（实时推送） |
| 服务器压力 | 高（频繁请求） | 低（单连接） |
| 资源消耗 | 高 | 低 |
| 实时性 | 秒级 | 毫秒级 |
| 双向通信 | 不支持 | 支持 |

#### 4.1.2 连接流程

客户端与券商服务端建立长连接的完整流程如下：

**第一步：用户登录**

用户通过客户端输入用户名和密码，向账户管理服务端发起登录请求。账户管理服务端验证用户身份后，生成包含用户ID、账户ID、权限列表、券商列表等信息的JWT Token，并返回给客户端。

**第二步：建立WebSocket连接**

客户端获取Token后，根据返回的券商地址列表，通过WebSocket协议与服务端建立TCP连接。连接建立后，客户端发送AUTH认证消息，携带Token进行身份验证。

**第三步：认证确认**

券商服务端接收到AUTH消息后，调用账户管理服务端验证Token有效性。验证通过后，返回AUTH_ACK消息，包含会话ID、用户信息、权限列表等，连接正式建立。

**第四步：交易通信**

连接建立后，客户端与服务端进入正常通信状态，可以进行下单、撤单、查询等交易操作。服务端也会主动推送订单状态变化、成交回报等信息。

**第五步：心跳保活**

连接建立后，客户端每隔30秒向服务端发送PING心跳消息。服务端响应PONG消息，确认连接活跃。若连续3次（90秒）未收到响应，则判定连接断开，触发重连机制。

#### 4.1.3 心跳保活机制

心跳保活机制用于检测连接是否存活，防止因网络波动或服务端异常导致的"死连接"问题。

**配置参数：**

| 参数 | 默认值 | 说明 |
|------|-------|------|
| HEARTBEAT_INTERVAL | 30秒 | 心跳发送间隔 |
| HEARTBEAT_TIMEOUT | 90秒 | 心跳超时阈值 |
| MAX_RECONNECT_ATTEMPTS | 5次 | 最大重连次数 |
| RECONNECT_INTERVAL | 5秒 | 初始重连间隔 |

**心跳状态机：**

```
┌────────────────────────────────────────────────────────────────────────────┐
│                         心跳状态转换                                        │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│   ┌─────────┐     发送PING      ┌─────────┐     超时90s     ┌─────────┐    │
│   │  正常   │─────────────────▶│  等待   │────────────────▶│  断开   │    │
│   │CONNECTED│◀─────────────────│  PONG   │◀────────────────│TIMEOUT  │    │
│   └─────────┘     收到PONG      └─────────┘   重试成功      └─────────┘    │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

**自动重连机制：**

当连接断开时，客户端自动执行重连，采用指数退避策略避免频繁重试：

| 重试次数 | 延迟计算 | 实际延迟 |
|---------|---------|---------|
| 1 | 5 × 2^0 = 5s | 5秒 |
| 2 | 5 × 2^1 = 10s | 10秒 |
| 3 | 5 × 2^2 = 20s | 20秒 |
| 4 | 5 × 2^3 = 40s | 30秒（上限） |
| 5 | 5 × 2^4 = 80s | 30秒（上限） |

重连成功后，客户端会自动同步断开期间的订单状态和持仓数据，确保数据一致性。

### 4.2 消息协议设计

#### 4.2.1 消息格式

所有WebSocket消息采用统一格式，便于解析和处理：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| type | string | 是 | 消息类型，标识消息用途 |
| seq | string | 是 | 消息序列号，用于消息追踪和响应匹配 |
| timestamp | long | 是 | 时间戳（毫秒），用于消息时效性验证 |
| data | object | 是 | 消息数据体，包含具体业务数据 |
| signature | string | 否 | 消息签名，用于安全验证 |

#### 4.2.2 消息类型分类

| 类别 | 消息类型 | 说明 |
|------|---------|------|
| 连接管理 | AUTH / AUTH_ACK / PING / PONG / DISCONNECT | 用于连接建立、认证、心跳、断开 |
| 交易操作 | ORDER_SUBMIT / ORDER_CANCEL / ORDER_MODIFY | 用于下单、撤单、改单操作 |
| 查询操作 | POSITION_QUERY / ORDER_LIST_QUERY / TRADE_HISTORY_QUERY / BALANCE_QUERY | 用于持仓、委托、成交、资金查询 |
| 推送消息 | ORDER_UPDATE / TRADE_UPDATE / POSITION_UPDATE / ALERT | 服务端主动推送的消息 |

#### 4.2.3 详细消息定义

**认证消息**

客户端连接后发送AUTH消息进行认证，包含Token和客户端标识信息。服务端验证通过后返回AUTH_ACK消息，包含会话ID、用户信息、权限列表、Token有效期等。

认证消息的成功响应表明连接已建立，用户可以进行后续交易操作。

**下单消息**

客户端发送ORDER_SUBMIT消息请求下单，包含股票代码、买卖方向、订单类型、价格、数量等参数。服务端处理后返回ORDER_SUBMIT_ACK消息，包含券商订单号、订单状态、处理结果等。

**撤单消息**

客户端发送ORDER_CANCEL消息请求撤单，通过订单号指定要撤销的委托。服务端返回ORDER_CANCEL_ACK消息，包含撤单结果和订单最终状态。

**查询消息**

客户端发送POSITION_QUERY消息查询持仓，发送ORDER_LIST_QUERY消息查询委托列表，发送BALANCE_QUERY消息查询资金余额。服务端返回对应的ACK消息，包含查询结果数据。

**推送消息**

服务端主动推送ORDER_UPDATE消息通知订单状态变化，推送TRADE_UPDATE消息通知成交回报，推送POSITION_UPDATE消息通知持仓变动，推送ALERT消息通知风险告警。

ConnectionManager是客户端的核心连接管理组件，负责与券商服务端建立和维护WebSocket连接。主要功能包括：

- WebSocket连接建立：通过WebSocket协议与服务端建立长连接
- 连接池管理：管理多个券商连接，支持切换
- 重连机制：连接断开时自动重连，采用指数退避策略
- 心跳维护：定时发送心跳消息，检测连接状态
- 加密存储：安全存储Token等敏感信息

**MessageDispatcher模块说明：**

MessageDispatcher负责接收和分发所有WebSocket消息，确保消息被正确处理。主要功能包括：

- 消息队列：缓存接收到的消息，支持异步处理
- 类型路由：根据消息类型分发到对应的处理器
- 响应匹配：将响应消息与请求消息关联，支持请求-响应模式
- 错误处理：捕获和处理消息处理过程中的异常

**TradingService模块说明：**

TradingService是交易业务的核心服务类，封装了所有交易相关的业务逻辑。主要功能包括：

- 下单操作：构造下单请求，发送并等待响应
- 撤单操作：构造撤单请求，发送并等待响应
- 改单操作：构造改单请求，修改委托价格和数量
- 查询操作：构造查询请求，获取持仓、委托、资金等信息

**APIGateway模块说明：**

APIGateway是券商服务端的入口层，负责处理所有客户端连接请求。主要功能包括：

- 协议解析：解析WebSocket消息，提取消息类型和数据
- 消息路由：将消息分发到对应的处理器
- 流量控制：限制单用户请求频率，防止滥用
- 熔断降级：当下游服务异常时，自动降级处理

**SessionManager模块说明：**

SessionManager负责管理所有客户端会话，是服务端状态管理的核心。主要功能包括：

- 会话管理：维护所有在线会话的映射关系
- 心跳监控：检测连接活跃状态，清理超时连接
- 空闲检测：识别长时间无活动的连接
- 会话存储：将会话数据存储到Redis，支持分布式部署

**TradingEngine模块说明：**

TradingEngine是交易处理的核心引擎，负责处理所有交易相关请求。主要功能包括：

- 订单验证：检查订单参数合法性，如价格范围、数量限制等
- 风险检查：执行交易风控规则，如仓位限制、资金限制等
- 订单路由：根据订单类型和规则，选择合适的处理路径
- 订单执行：调用券商适配器执行订单

**BrokerAdapter模块说明：**

BrokerAdapter是券商对接的适配层，负责与券商本地系统交互。主要功能包括：

- API调用：调用券商提供的交易API
- 协议转换：将内部消息格式转换为券商要求的格式
- 错误映射：将券商返回的错误转换为统一错误码
- 超时处理：处理券商API调用超时情况
